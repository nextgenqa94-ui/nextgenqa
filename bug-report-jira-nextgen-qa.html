Bug Report Template · All Fields Explained · Sample JIRA Bug Cards
Bug Report — All Fields
🆔 Bug ID
Unique identifier. e.g. BUG-101, LOGIN-005, PAYMENT-003. Auto-generated in JIRA.
📌 Bug Title / Summary
Short, clear description. Format: [Module] – What is wrong. e.g. "Login – Valid user gets 'Invalid credentials' error"
📋 Description
Detailed explanation of the bug — what happened, when it happens, how frequently.
🔁 Steps to Reproduce
Numbered step-by-step actions to see the bug. Must be clear so developer can reproduce it exactly.
✅ Expected Result
What SHOULD happen as per requirements or design. Be specific and measurable.
❌ Actual Result
What ACTUALLY happened — the bug behavior observed. Copy exact error messages if any.
🔴 Severity
How badly does the bug impact functionality? Critical / Major / Moderate / Minor / Trivial. Set by Tester.
⚡ Priority
How urgently should it be fixed? P1-High / P2-Medium / P3-Low. Set by Product Manager / Team Lead.
🌐 Environment
OS, Browser, Version, Device. e.g. Windows 11, Chrome 120, QA Environment v2.3
📎 Attachments
Screenshot, screen recording, log files, HAR file. Visual proof is essential!
📦 Build Version
Version/build where bug was found. e.g. v2.3.1, Build #1045
🏷️ Labels / Tags
Keywords for searching. e.g. login, regression, payment, mobile

Severity Levels · Priority Levels · Severity vs Priority Matrix · Classic Combinations
Severity Levels — Set by Tester
🔴 Critical (S1)
Application completely crashes or data is lost. No workaround available. Testing must STOP. e.g. Payment fails, app crashes on login, database error.
🟠 Major (S2)
Major feature broken. Workaround exists but is difficult. Core functionality impacted. e.g. Search returns wrong results, user cannot upload files.
🟡 Moderate (S3)
Partial feature failure. Easy workaround available. Some functionality affected. e.g. Filter not working on one page, wrong date format shown.
🟢 Minor (S4)
Small UI issue or cosmetic problem. Does not affect functionality. e.g. Button color is wrong, text is misaligned, spelling error.
🔵 Trivial (S5)
Very minor visual issue. Barely noticeable. e.g. Extra space in footer, tooltip text slightly off-center.
Priority Levels — Set by Manager/PO
⚡ P1 — Critical / Immediate
Fix NOW. Cannot ship without this fix. Show-stopper bug. Drop everything else. e.g. Production is down, data corruption.
🔥 P2 — High
Fix in current sprint. Important feature is broken. High business impact. e.g. Registration not working, wrong price displayed.
📌 P3 — Medium
Fix in next sprint. Feature is degraded but workaround available. e.g. Export CSV has wrong column order.
📋 P4 — Low
Fix when time permits. Minor issue. Does not affect user experience significantly. e.g. Hover animation jerky on old browser.

Bug Lifecycle — All States
New
Bug discovered and logged by QA tester for first time.
Assigned
Bug reviewed and assigned to developer by Team Lead.
Open / In Progress
Developer is actively working on fixing the bug.
Fixed / Resolved
Developer has fixed the bug and moved it for testing.
Ready for Retest
Fix is deployed to QA environment. Tester must retest.
Verified / Closed
Tester confirms fix works. Bug is officially closed.
Reopened
Bug still exists after fix. Sent back to developer.
Deferred
Valid bug but postponed to next release. Low priority or resource constraint.
Not a Bug / Rejected
Developer says it works as designed. Discussed with team for final decision.
Duplicate
Same bug already reported by another tester. Linked to original bug.
Cannot Reproduce
Developer cannot reproduce the bug. Tester must provide more details / steps.

Steps to Reproduce — Tips
✅ Be Specific
Write exact values used — URL, username, button name, exact input.
✅ Number Every Step
1. Open URL → 2. Enter data → 3. Click button. Each action = one step.
✅ Mention Preconditions
State what needs to be set up first. e.g. "User must be logged in", "Cart must have items".
❌ Don't Skip Steps
Developer must be able to reproduce EXACTLY what you did. Missing steps = unreproducible bug.

Interview Q&A — Part 1
Q: What is a Bug Report?
Document that describes a defect — includes bug ID, title, steps to reproduce, expected result, actual result, severity, priority, environment, and attachments.
Q: What is Bug Life Cycle?
States a bug goes through: New → Assigned → Open → Fixed → Retest → Verified/Closed. Can also go to Reopened, Deferred, Rejected, or Duplicate.
Q: What is Severity vs Priority?
Severity = technical impact (set by QA). Priority = business urgency (set by Manager). Logo missing is Minor Severity but High Priority — brand image matters.
Q: What is Reopened status?
Bug is marked Reopened when QA verifies the fix but the bug still exists. Sent back to developer for another fix attempt.
Q: What is Deferred status?
Bug is valid but postponed to next release due to time constraints, low priority, or pending dependency. Not rejected — will be fixed later.
Q: What is a Duplicate Bug?
When the same bug has already been reported by another tester. Mark as Duplicate and link to the original bug ID.
Q: What is "Cannot Reproduce"?
Developer cannot replicate the bug with given steps. QA must add more details, environment info, or screen recording to help reproduce it.
Interview Q&A — Part 2
Q: What makes a good bug report?
Clear title, exact steps to reproduce, specific expected & actual results, severity/priority set correctly, environment details, screenshot or video attached.
Q: What is JIRA?
Project management and bug tracking tool by Atlassian. Used to create, track, assign, and manage bugs and tasks across sprints and releases.
Q: What is JQL?
JIRA Query Language — used to search/filter issues. e.g. issuetype = Bug AND priority = Critical AND status = Open
Q: What is a Critical bug example?
App crashes on login, payment fails, data is lost, database error, complete feature not working with no workaround available.
Q: How do you write Steps to Reproduce?
Number each step clearly. Include exact URL, input values, button names, preconditions. Each action = one step. Developer must reproduce exactly.
Q: What is Defect Leakage?
When a bug escapes testing and is found by end user in production. Indicates missing test coverage. Measured as (bugs in prod / total bugs) × 100%.
Q: What attachments should a bug have?
Screenshot showing the issue, screen recording of the bug happening, browser console logs, network HAR file for API issues, error log files.
