SDLC — Software Development Life Cycle · STLC — Software Testing Life Cycle
SDLC Phases
1
Requirement Gathering
Collect & analyze business requirements from client
2
System Design
HLD (High Level) + LLD (Low Level) design documents
3
Implementation / Coding
Developers write the actual source code
4
Testing
QA team tests the developed software
5
Deployment
Software released to production/live environment
6
Maintenance
Bug fixes, updates and enhancements post-release
STLC Phases
1
Requirement Analysis
QA studies & reviews requirements — identifies testable items
2
Test Planning
Scope, strategy, resources, timeline, tools decided
3
Test Case Development
Write test cases, test scripts, test data preparation
4
Test Environment Setup
Configure hardware, software, network for testing
5
Test Execution
Execute test cases, log defects, retest fixed bugs
6
Test Closure
Test summary report, lessons learned, sign-off
SDLC Models
Waterfall Model
Sequential — each phase completes before next begins. No going back. Best for fixed requirements.
V-Model
Each dev phase has a corresponding test phase. Testing starts early. Verification + Validation.
Agile Model
Iterative sprints (2–4 weeks). Continuous delivery. Flexible to changes. Most popular today.
Spiral Model
Risk-driven. Combines Waterfall + Prototyping. Multiple loops. Good for large risky projects.
Incremental Model
Software built in increments. Each increment adds functionality. Partial working product early.
RAD Model
Rapid Application Development. Prototype-based. Fast delivery. Works with active user feedback.

Entry & Exit Criteria
Entry Criteria
Conditions that MUST be met BEFORE testing starts. e.g. Build is deployed, test environment is ready, test data is available.
Exit Criteria
Conditions that MUST be met to STOP testing. e.g. All test cases executed, 95% pass rate, no critical bugs open.
Types of Testing — Functional · Non-Functional · Structural · Change-Based
Functional Testing Types
Unit Testing
Testing individual units/components in isolation. Done by developers.
Integration Testing
Testing interaction between integrated modules. STBI (Top-Down, Bottom-Up, Sandwich).
System Testing
Testing complete integrated system against requirements.
UAT (User Acceptance)
End users test software to verify it meets business needs before go-live.
Smoke Testing
Preliminary testing — checks if build is stable enough for further testing. "Build Verification Test".
Sanity Testing
Narrow regression after bug fix — verifies specific functionality works correctly.
Regression Testing
Re-testing after code change to ensure existing functionality still works.
Re-testing
Re-executing failed test cases after defect is fixed to confirm it's resolved.
Non-Functional Testing Types
Performance Testing
Evaluates speed, responsiveness and stability. Parent of Load, Stress, Spike testing.
Load Testing
Test behavior under expected normal load. Find max operating capacity.
Stress Testing
Test beyond normal capacity to find breaking point. What happens when overloaded?
Spike Testing
Sudden large increase in load. How does system handle sudden traffic bursts?
Security Testing
Find vulnerabilities, threats, risks. SQL injection, XSS, Authentication bypass.
Usability Testing
How user-friendly is the application? Ease of use, intuitiveness, learnability.
Compatibility Testing
Test across different OS, browsers, devices and screen sizes.
Reliability Testing
Can system perform its intended function without failure for specific time?
Scalability Testing
Can system handle growing workload gracefully? Vertical & horizontal scaling.
Recovery Testing
How well does system recover from crashes, failures, hardware/network issues?

More Testing Types
Exploratory Testing
Simultaneous learning, test design and execution. No predefined scripts. Tester's intuition.
Ad-hoc Testing
Unplanned, informal testing without documentation. Random testing to find unexpected bugs.
Alpha Testing
Internal testing by developer team before release. Done at developer's site.
Beta Testing
External testing by real users before final release. Done at customer's site.
Monkey Testing
Random input testing without any test cases. Test like a monkey — anything goes!

Testing for specific region — language, date format, currency, cultural content.
Globalization Testing

Complete workflow testing from start to finish. Simulates real user scenarios.

Test Case Components
Test Case ID
Unique identifier. e.g. TC_LOGIN_001, TC_PAYMENT_002
Test Case Title
Short description of what is being tested. Clear and specific.
Preconditions
Conditions that MUST be true before executing. e.g. User must be registered.

Step-by-step actions to perform. Clear, concise, numbered.
Test Data
Input values used for testing. e.g. Username: admin, Password: admin123
Expected Result
What SHOULD happen after executing steps. Be specific!
Actual Result
What ACTUALLY happened when test was executed.
Status
Pass / Fail / Blocked / Not Executed / In Progress
Priority
High / Medium / Low — based on business importance
Test Type
Positive / Negative / Boundary

Test Design Techniques — BVA · Equivalence Partitioning · Decision Table · State Transition
Equivalence Partitioning (EP)
Definition
Divide input data into PARTITIONS where each value in a partition is expected to behave the same. Test ONE value from each partition.
Example: Age field (18–60 valid)

Boundary Value Analysis (BVA)
Definition
Test values AT THE EDGES of partitions. Bugs are most likely to occur at boundaries. Used with EP.

Error Guessing
Definition
Tester uses experience and intuition to GUESS where errors might be. No formal technique.
Common Error Guess Areas
Empty input, zero values, negative numbers, very long strings, special characters, null values, duplicate data, max file size.
Test Coverage Techniques
Statement Coverage
Every line of code executed at least once. Basic coverage metric.
Branch Coverage
Every branch (if/else) executed — both TRUE and FALSE paths covered.
Path Coverage
Every possible path through the code is tested. Most thorough but expensive.
Condition Coverage
Every condition in decision tested for both true and false outcome.
Use Case Testing
Use Case Testing
Test based on use cases — real user scenarios from start to finish. Tests end-to-end workflows from user perspective.
Pairwise Testing
Pairwise / All-pairs
Tests every possible PAIR of input values at least once. Reduces test cases while maintaining good coverage.

Top Interview Questions & Answers — Must Know for Every QA Fresher
Must-Know Interview Q&A — Part 1
Q: What is Software Testing?
Process of evaluating software to find bugs, ensure it meets requirements and works as expected before release.
Q: What is a Bug/Defect?
Any deviation between expected result and actual result. A mistake in code that causes incorrect behavior.
Q: What is Test Plan?
Document that describes scope, approach, resources and schedule of testing activities for a project.
Q: Difference between Error, Bug, Failure?
Error = Human mistake in code. Bug = Defect in code (result of error). Failure = System doesn't work as expected (result of bug).
Q: What is Quality Assurance vs Quality Control?
QA = Process-oriented, prevent defects (SDLC). QC = Product-oriented, find defects (actual testing).
Q: What is Test Environment?
Setup of hardware, software, OS, network and tools required to execute test cases.
Q: What is Traceability Matrix?
Document mapping requirements to test cases. Ensures no requirement is missed during testing.
Must-Know Interview Q&A — Part 2
Q: What is Boundary Value Analysis?
Testing values at the edges of valid/invalid input ranges. Bugs are most likely at boundaries — e.g. 17, 18, 19 for age min of 18.
Q: What is Equivalence Partitioning?
Dividing input data into valid and invalid partitions. Testing one value from each partition covers the entire partition.
Q: What is Regression Testing?
Re-testing existing functionality after code changes/bug fixes to ensure nothing is broken. Can be automated.
Q: What is UAT?
User Acceptance Testing — end users test the software to verify it meets business requirements before go-live.
Q: What is the difference between Smoke and Sanity?
Smoke = Wide & shallow check of entire build stability. Sanity = Narrow & deep check of specific bug fix or feature.
Q: What makes a good test case?
Clear title, specific steps, expected result, test data, and covers positive, negative and boundary scenarios.
Q: What tools do you know?
JIRA (bug tracking), Selenium (automation), Playwright, Postman (API), TestNG, JMeter (performance).
Must-Know Interview Q&A — Part 3
Q: What is Severity vs Priority?
Severity = Bug's impact on functionality (set by tester). Priority = Order of fixing (set by manager). High Severity ≠ High Priority always.
Q: What is Exploratory Testing?
Unscripted testing where tester simultaneously learns, designs and executes tests using experience and intuition.
Q: What is End-to-End Testing?
Testing complete workflow from start to finish simulating real user behavior. e.g. Login → Add to cart → Checkout → Payment → Confirmation.
Q: What is Alpha vs Beta Testing?
Alpha = Internal testing by dev team at developer's site. Beta = External testing by real users at customer's location.
Q: How do you report a bug?
Bug ID, Title, Module, Environment, Steps to Reproduce, Expected Result, Actual Result, Severity, Priority, Screenshot/Video.
Q: What is Test Closure?
Last phase of STLC — activities include creating test summary report, lessons learned document, and formal sign-off from stakeholders.
Q: What is Defect Leakage?
When a bug escapes testing and is found by end user in production. Indicates gap in test coverage.
